# Breadth-First Search

"How many introductions connect you to anyone on the platform?" "What is the fewest moves to
solve this puzzle state?" "Which servers go down if this one fails?" All three are the same
question — *what is reachable, and how close is it* — and BFS is the answer to all of them.

## Rings, not tunnels

Picture dropping a stone in water: the wave reaches everything one unit away, then everything
two units away, never skipping a ring. BFS explores a graph exactly like that — it finishes
distance 1 completely before touching distance 2.

The machinery is a **queue** — first in, first out. That single choice produces the ring-by-ring
order: nodes discovered earlier are processed earlier, so nearer nodes always come first.

## The algorithm

1. Put the start node in the queue and mark it **discovered**.
2. While the queue is not empty:
   - Dequeue the front node `u`.
   - For each neighbor `w` of `u`: if `w` is not yet discovered, mark it and enqueue it.
   - Mark `u` **visited** (fully processed).

Two states matter, and the visualization colors them differently on purpose:

- **Frontier (cyan):** discovered and queued, but neighbors not yet examined.
- **Visited (violet):** dequeued and fully processed.

<Callout kind="warning">
  Mark nodes discovered **when they are enqueued**, not when they are dequeued. Mark-on-dequeue
  lets the same node enter the queue multiple times through different neighbors — correct
  output, but the queue can blow up to O(E) entries and the run slows badly on dense graphs.
</Callout>

<Callout kind="insight">
  **Shortest paths for free.** Because BFS finishes ring k before starting ring k+1, the first
  time it reaches a node is via a minimum-edge path. In any unweighted graph, "BFS discovery
  order" *is* "shortest path order" — no extra bookkeeping needed. (Weighted graphs need
  Dijkstra, which is BFS with a priority queue.)
</Callout>

## Complexity

| Aspect | Cost | Why |
| --- | --- | --- |
| Time | `O(V + E)` | each node enqueued once, each edge examined once (twice if undirected) |
| Space | `O(V)` | the discovered set, plus a queue that peaks at one ring |

`O(V + E)` is as good as it gets — you cannot answer reachability without looking at every
edge at least once.

## Summary

- BFS explores in rings using a queue; nearer nodes are always finished first.
- Mark discovered on **enqueue** — the classic correctness-adjacent performance bug.
- First arrival = shortest path in unweighted graphs; no extra work required.
- Cost `O(V + E)` time, `O(V)` space — optimal for reachability questions.
