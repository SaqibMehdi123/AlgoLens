# Insertion Sort

Insertion sort is how you sort a hand of playing cards: pick up one card at a time and slide it left
until it sits in the right place among the cards you've already sorted. It is `O(n²)` like selection
sort — but with two superpowers that make it the sort production code actually reaches for on small
or nearly-sorted inputs.

## Slide each element into place

<Viz algo="insertion-sort" frame={12} caption="The active element (amber) slides left past everything larger than it; the mint prefix stays sorted at every step." />

Maintain a sorted prefix on the left. Take the next element and swap it leftward, one adjacent step
at a time, until the element to its left is no longer larger. The prefix is sorted after *every*
insertion — that running invariant is what makes the algorithm easy to reason about.

## The algorithm

1. For `i` from `1` to `n − 1` (the prefix `a[0]` is trivially sorted):
   - Let `j = i`.
   - While `j > 0` and `a[j − 1] > a[j]`: swap them and decrement `j`.
2. Each pass grows the sorted prefix by one.

<Callout kind="insight">
  **Adaptive: O(n) on nearly-sorted data.** If each element is close to its final spot, the inner
  `while` barely runs — on an already-sorted array it does *one* comparison per element and zero
  swaps, so the whole sort is `O(n)`. This is why hybrid sorts (Timsort, introsort) switch to
  insertion sort for small or almost-sorted runs.
</Callout>

<Callout kind="warning">
  **Online and stable — don't break it.** Insertion sort can sort data as it arrives (it doesn't
  need the whole array up front), and it is **stable** because it only swaps strict inversions
  (`>`, never `≥`). Changing that `>` to `≥` destroys stability for no benefit.
</Callout>

## Complexity

| Case | Time | When |
| --- | --- | --- |
| Best | `O(n)` | already sorted — one compare per element |
| Average | `O(n²)` | about n²/4 swaps |
| Worst | `O(n²)` | reverse-sorted — every element slides to the front |
| Space | `O(1)` | sorts in place |

The best case is the headline. Unlike selection sort — `O(n²)` even on sorted input — insertion sort
*rewards* pre-existing order, degrading gracefully from `O(n)` toward `O(n²)` as disorder increases.

## In interviews & contests

- **Reach for it** for small arrays (n ≲ 16), nearly-sorted data, or streaming input. It is the
  small-array base case *inside* quicksort and mergesort implementations.
- **Interviewers probe:** "When does an O(n²) sort beat an O(n log n) one?" — small or nearly-sorted
  n, where insertion sort's tiny constant and best-case O(n) win. Also: stable? (yes); adaptive? (yes).
- **Contest mapping:** rarely your main sort, but its invariant ("prefix always sorted") is the
  mental model behind inserting into sorted structures and patience-sorting ideas.

## Summary

- Grow a sorted prefix; slide each new element left past larger ones (one swap per inversion).
- Best case `O(n)` on sorted/nearly-sorted data, `O(n²)` average and worst, `O(1)` space.
- Stable, online, and adaptive — the reasons hybrid sorts use it for small runs.
