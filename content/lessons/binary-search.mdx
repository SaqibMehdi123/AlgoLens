# Binary Search

Think of the number-guessing game: I pick a number between 1 and 100, you guess, I say
"higher" or "lower". Nobody scans 1, 2, 3… — you guess 50, then 75, then 62. Seven guesses
always suffice, because each one halves what is left.

Binary search is that strategy, written down precisely. It needs one thing in exchange:
**the array must be sorted.**

## Watch the window shrink

The algorithm keeps a window `lo..hi` where the target must live, if it exists. Each step it
examines the middle element and throws away the half that cannot contain the target:

algorithm's **invariant**, and it is what makes the code trustworthy.

## The algorithm

1. Start with `lo = 0`, `hi = n − 1`.
2. While `lo ≤ hi`:
   - `mid = ⌊(lo + hi) / 2⌋`
   - If `a[mid]` equals the target — found, return `mid`.
   - If `a[mid]` is **less** than the target, the answer can only be right of mid: `lo = mid + 1`.
   - Otherwise it can only be left of mid: `hi = mid − 1`.
3. If the loop exits, the window is empty: the target is not in the array.

Two details cause most real-world bugs:

<Callout kind="warning">
  **Off-by-one traps.** The loop condition is `lo ≤ hi` (not `<`), and the updates must skip
  `mid` (`mid + 1` / `mid − 1`). Get either wrong and the loop can run forever on a two-element
  window. When in doubt, step the visualization backward and check the invariant at each frame.
</Callout>

<Callout kind="insight">
  **Sorted is the contract.** On unsorted data binary search returns garbage *silently* — it
  still terminates, it just discards the wrong half. If you cannot guarantee order, sort first
  (O(n log n)) or use a hash set (O(1) lookups) instead.
</Callout>

## Complexity

| Case | Time | Why |
| --- | --- | --- |
| Best | `O(1)` | first mid is the target |
| Average | `O(log n)` | each compare halves the window |
| Worst | `O(log n)` | window shrinks 8 → 4 → 2 → 1 |
| Space | `O(1)` | three integer variables |

The base of the logarithm does not matter (it is a constant factor); what matters is that
1,000,000 elements cost about 20 compares, and doubling the data adds **one** compare.

## Summary

- Binary search maintains a window that provably contains the target, and halves it each step.
- `lo ≤ hi`, `mid + 1`, `mid − 1` — the three places bugs live.
- Sorted input is non-negotiable; violations fail silently.
- Cost: `O(log n)` time, `O(1)` space — doubling the data adds one compare.
