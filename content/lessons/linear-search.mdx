# Linear Search

Before you can earn the speed of binary search, you have to know what it beats. Linear search is
the baseline every interviewer assumes you can write in your sleep: walk the array, return the
first match. It makes **no assumptions** about the data — which is both its weakness and, sometimes,
its superpower.

## Walk until you hit it

The algorithm is a single loop. Index `0`, index `1`, … compare each element to the target; the
moment one matches, return its position. If the loop runs off the end, the target was not there.

## The algorithm

1. For `i` from `0` to `n − 1`:
   - If `a[i]` equals the target, return `i`.
2. Return `−1` — the loop finished without a match.

<Callout kind="insight">
  **Unsorted is fine.** Linear search is the only search that needs no preprocessing. On data you
  will scan exactly once, sorting first (O(n log n)) just to enable binary search is a net loss —
  the single scan is already O(n).
</Callout>

<Callout kind="warning">
  **The sentinel trick.** Some solutions append the target as a sentinel at `a[n]` so the loop can
  drop its bounds check (`while a[i] != target`). It shaves one comparison per iteration — a
  micro-optimization, but a classic interview talking point (Knuth, *TAOCP* Vol. 3).
</Callout>

## Complexity

| Case | Time | When |
| --- | --- | --- |
| Best | `O(1)` | target is the first element |
| Average | `O(n)` | target is somewhere in the middle |
| Worst | `O(n)` | target is last, or absent |
| Space | `O(1)` | one index variable |

The takeaway interviewers want: linear search is optimal when the data is **unsorted and traversed
once**. The instant you search the same array repeatedly, preprocess it — sort for binary search,
or build a hash set for O(1) lookups.

## In interviews & contests

- **Reach for it** when n is tiny, the array is unsorted and used once, or a hash set is overkill.
  Don't apologize for O(n) when the input is genuinely unstructured.
- **Interviewers probe:** "Can you do better?" → only with a precondition (sorted → binary search)
  or extra space (hash set). Saying that trade-off out loud is the point.
- **Contest mapping:** with n ≤ ~10⁶ and a single pass, O(n) clears a 1-second limit comfortably;
  repeated queries push you to sort + binary search, or a frequency array.

## Summary

- Linear search scans front to back and returns the first match; no preconditions.
- `O(n)` time, `O(1)` space — optimal for unsorted, single-pass searches.
- Repeated queries are the signal to preprocess (sort + binary search, or a hash set).
