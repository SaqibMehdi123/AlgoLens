# Big-O Notation

Two functions can both be "correct" and still behave completely differently in production.
One returns in a millisecond at a million elements; the other locks up your server. The
difference is rarely visible in the code review — it is visible in how the work *grows*.

Big-O is the tool for seeing that growth before you ship.

## Count steps, not seconds

Wall-clock time is a bad measure: it changes with your laptop, the JIT, even the weather in
your data center. What does not change is **how many basic operations** an algorithm performs
for an input of size `n`.

Watch linear search scan an array. Every element it touches is one unit of work — the bars
light up one by one, left to right:

If the target sits at the end — or is missing — the scan touches all `n` elements. Double the
array, double the work. That proportionality is the whole idea: linear search is **O(n)**.

Now compare the same search done by halving (you will meet this algorithm properly in the next
lesson). It throws away half the remaining window at every step:

Eight elements need at most 3 halvings. A *million* elements need about 20. That is **O(log n)**
— growth so slow it barely registers.

## The formal idea, minus the ceremony

`f(n)` is `O(g(n))` if, past some input size, `f` is bounded above by a constant multiple of
`g`. In practice that means three working rules:

1. **Drop constants.** `3n + 40` is `O(n)`. Constants matter for benchmarks, not for growth.
2. **Keep only the dominant term.** `n² + n + 1000` is `O(n²)` — for large `n` the square is
   all that matters.
3. **Nesting multiplies, sequence takes the max.** A loop inside a loop over the same input is
   `n · n = n²`. Two loops one after another are `n + n`, still `O(n)`.

## The growth classes you will actually meet

| Class | Name | Feel at n = 1,000,000 |
| --- | --- | --- |
| `O(1)` | constant | instant |
| `O(log n)` | logarithmic | ~20 steps |
| `O(n)` | linear | a million steps — fine |
| `O(n log n)` | linearithmic | ~20 million — fine |
| `O(n²)` | quadratic | a trillion — minutes to hours |
| `O(2ⁿ)` | exponential | the universe ends first |

The cliff between `O(n log n)` and `O(n²)` is the one that bites real systems: it is the
difference between merge sort and bubble sort, between a join with an index and one without.

## Summary

- Measure **growth in operations**, not seconds: it is the property that survives hardware.
- Drop constants, keep the dominant term; nesting multiplies, sequence takes the max.
- Learn the six classes in the table — they cover nearly everything you will write.
- When you are unsure, *count what the visualization actually does*: the step counter in the
  player is an operation count.
