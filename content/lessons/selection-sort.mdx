# Selection Sort

Selection sort is how most people sort a hand of cards without thinking about it: find the smallest
card, put it first, then find the smallest of what's left, and repeat. It is rarely the right choice
in production ‚Äî but it is the cleanest way to *see* what "sorted" means, and it sets up every better
sort by showing exactly what they improve on.

## Find the minimum, place it, repeat

<Viz algo="selection-sort" frame={10} caption="Amber bars are the current scan for the minimum; mint bars are locked in their final sorted position." />

The array splits into two regions: a sorted prefix on the left (growing) and an unsorted suffix on
the right (shrinking). Each pass scans the suffix for its minimum and swaps it to the front of the
suffix ‚Äî extending the sorted region by one.

## The algorithm

1. For `i` from `0` to `n ‚àí 2`:
   - Let `min = i`.
   - For `j` from `i + 1` to `n ‚àí 1`: if `a[j] < a[min]`, set `min = j`.
   - Swap `a[i]` and `a[min]`.
2. After pass `i`, `a[0..i]` holds the `i + 1` smallest elements, sorted.

<Callout kind="insight">
  **Exactly n ‚àí 1 swaps.** Selection sort writes to the array at most once per pass ‚Äî fewer writes
  than any other O(n¬≤) sort. On hardware where a *write* costs far more than a *read* (e.g., flash
  memory), that property occasionally makes it the right tool.
</Callout>

<Callout kind="warning">
  **The comparisons never shrink.** Even on an already-sorted array, selection sort scans the entire
  suffix every pass ‚Äî so it is O(n¬≤) in the *best* case too. It is also **not stable**: the
  long-distance swap can reorder equal keys. Insertion sort fixes both of these.
</Callout>

## Complexity

| Case | Time | Why |
| --- | --- | --- |
| Best | `O(n¬≤)` | the inner scan runs fully regardless of order |
| Average | `O(n¬≤)` | about n¬≤/2 comparisons |
| Worst | `O(n¬≤)` | same scan, every pass |
| Space | `O(1)` | sorts in place |
| Swaps | `O(n)` | at most one per pass |

The n¬≤/2 comparisons come from summing the shrinking inner loop: (n‚àí1) + (n‚àí2) + ‚Ä¶ + 1 = n(n‚àí1)/2.
That arithmetic series is the fingerprint of every quadratic sort.

## In interviews & contests

- **Reach for it** almost never for speed ‚Äî but name it when asked for the sort with the **fewest
  writes**, or to demonstrate the sorted/unsorted-region invariant cleanly.
- **Interviewers probe:** "Is it stable?" (no) and "Best case?" (still O(n¬≤) ‚Äî the fact that
  separates it from insertion sort).
- **Contest mapping:** any O(n¬≤) sort handles n ‚â≤ 10‚Å¥; beyond that you need an O(n log n) sort (or
  the library built-in). In real contests you call the built-in ‚Äî selection sort is a teaching tool.

## Summary

- Two regions ‚Äî sorted prefix, unsorted suffix; each pass moves one minimum across the boundary.
- `O(n¬≤)` comparisons in **every** case (no early exit), `O(1)` space, only `O(n)` swaps.
- Not stable; best case is still quadratic ‚Äî both reasons to prefer insertion sort on small inputs.
